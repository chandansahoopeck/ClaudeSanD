A working reference for how code moves from a feature branch to a running pod: the flow, the patterns underneath it, what's genuinely solid, what's fragile, and the exact lines of config worth rewriting.

Stage 00 · The shape of the system

Two job types, chained, not one pipeline

Every service runs on two separate Jenkins jobs: a Multibranch Pipeline that validates and publishes, and a Freestyle job that deploys. They're connected by a Jenkins upstream trigger, not a single unified Jenkinsfile — worth naming, because it shapes almost every trade-off below.

00 / 08

→
triggerbuild after upstream stable
→
Job BFreestyle CD job
→
targeton-prem Kubernetes
Jenkins Bitbucket JFrog Artifactory Docker Kubernetes (on-prem) Job-chaining trigger Freestyle, not Jenkinsfile, for CD
Done well

Clean separation of concerns between "did the code pass" and "is it deployed." A CI failure can never accidentally trigger a deploy, because the CD job only fires on a stable upstream build.

Worth fixing

Because CD lives in a Freestyle job instead of a Jenkinsfile, its logic isn't versioned in git the way CI's is. Every environment's deploy job is a hand-edited copy of the last one — see Env Config.

Stage 01 · Source control

Branching & protection rules

A GitFlow-lite model: short-lived feature branches merge into a protected develop, which promotes to a protected release. Nothing lands without review.

01 / 08
workfeature/*
→
bugfixbugfix/*
→
integrationdevelop
→
production-trackrelease

Pros

  • Feature isolation keeps in-flight work from blocking others — dozens of concurrent branches coexist cleanly.
  • Protected develop/release means the deploy targets are never touched by a stray direct push.
  • A single required approval keeps velocity high for a small team.

Cons

  • No documented hotfix path distinct from bugfix/* — unclear how an urgent prod fix skips the queue.
  • Named approvers (not a role/group) create a two-person bus factor on every merge.
  • Long-lived feature branches drift further from develop the longer review takes.

Use cases this fits

  • Small, trusted team shipping multiple services in parallel.
  • Environments where every change needs an audit trail for compliance.
  • Teams that deploy dev/test continuously but gate prod manually.
Stage 02 · Continuous Integration

PR validation — same pipeline, branch-conditional behavior

One Jenkinsfile runs on every branch a Multibranch Pipeline discovers. On a PR branch it validates only; on develop, the same stages actually publish the image. It's one pipeline wearing two hats, not two separate pipelines.

Trigger: PR created / branch push Validation-only on PR branches Publish-on-merge via same Jenkinsfile Avg run ≈ 1m 18s (agent-mail-service/develop)
Jenkinsfile — declarative, shape observed in Blue Ocean stage view
// Same file runs for both PR branches and develop —
// stage behavior branches on env.BRANCH_NAME
pipeline {
  agent any
  stages {
    stage('Checkout SCM') { steps { checkout scm } }

    stage('Lint') {
      steps { sh 'flake8 --diff $(git diff --name-only origin/develop...)' }
    }

    stage('Unit Tests') { steps { sh 'pytest -q' } }

    stage('Docker Build') {
      steps { sh 'docker build -t $APP_NAME:$BUILD_NUMBER .' }
    }

    stage('Smoke Test') { steps { sh './scripts/smoke.sh' } }

    stage('Push to Artifactory') {
      when { branch 'develop' }  // ← the branch-conditional hinge
      steps { sh 'docker push $ARTIFACTORY/$APP_NAME:$BUILD_NUMBER' }
    }
  }
}

Pros

  • One source of truth for build logic — no drift between "how PR builds work" and "how the real image gets built."
  • Docker image built during PR validation never gets published, so a broken PR can't pollute Artifactory.
  • Multibranch auto-discovery means new feature branches get CI for free, no manual job setup.

Cons

  • The branch-conditional publish step is implicit — easy to misread the docs as "three separate pipelines" (they aren't).
  • No visible integration/contract test stage between unit tests and smoke tests.
  • Smoke tests run against a locally built image, not the pushed one — a rebuild step in between could theoretically diverge.

Use cases this fits

  • Services with a single deployable artifact per repo.
  • Teams wanting fast PR feedback without provisioning separate validation infra.
  • Any repo where "tested" and "published" must never be the same event.
Done well

Reusing one Jenkinsfile across PR and merge events is the right instinct — it's the reason your Docker image and your tested code can never disagree with each other.

Stage 03 · Configuration

Environment config — the real gap between doc and disk

The design intent is dynamic resolution per environment. What's actually running is a hardcoded block at the top of each Freestyle CD job's shell step — one job per environment, values duplicated by hand.

03 / 08
As it runs today — hardcoded per job
agent-mail-service-dev-deployment-onmerge
# ============================
# Hard-coded configuration
# ============================
app_name="agent-mail-service"
target_build_env="dev"
kubectl_namespace="it-ex-automation-dev"
kubectl_context="svc-GSS-RPA-flash1.kube-dev"
Proposed — one job, config resolved by lookup
envs.yaml + one parameterized job
# envs.yaml — checked into the repo, diffable, reviewable
dev:
  namespace: it-ex-automation-dev
  context: svc-GSS-RPA-flash1.kube-dev
test:
  namespace: it-ex-automation-test
  context: svc-GSS-RPA-flash1.kube-test

# build step becomes:
cfg=$(yq ".${TARGET_ENV}" envs.yaml)
python -m scripts.deploy_to_k8s \
  --app "$APP_NAME" --env "$TARGET_ENV" \
  --namespace "$(echo "$cfg" | yq .namespace)" \
  --context   "$(echo "$cfg" | yq .context)"
One Freestyle job per service per environment Fresh venv every run requirements.txt installed on every build deploy_to_k8s.py does the actual kubectl work

Pros

  • Dead simple to read — anyone can open the job and see exactly what it'll deploy where.
  • Ephemeral venv per run means no dependency drift between builds.
  • Isolation: one env's job can't accidentally deploy to another namespace by config-merge error.

Cons

  • Adding an environment means copy-pasting a Jenkins job and hand-editing four strings — no compiler or reviewer catches a typo.
  • No single source of truth for namespace/context naming — a naming-convention change means editing N jobs.
  • Config isn't in git, so there's no diff/blame history for "when did dev's namespace change."

Use cases this fits

  • A small, fixed number of environments (dev/test/prod) that rarely change shape.
  • Teams prioritizing "can read the job and know exactly what happens" over DRY-ness.
Highest-priority fix

Of everything in this guide, this is the one worth doing first: it's low-risk (the deploy script itself doesn't change, only where its inputs come from) and it removes the most realistic failure mode — a hand-edit typo in a rarely-touched job.

Stage 04 · Governance

Code review, approval, and the post-merge build

Review checks business logic, standards, performance, and security. Once merged, the full codebase is linted and the image is built and pushed — no tests re-run, since they already ran on the PR.

04 / 08
gate1+ approval required
→
mergedevelop / release
→
full lintwhole codebase
→
publishimage → Artifactory

Pros

  • No redundant test run post-merge — saves pipeline minutes on every merge.
  • Full-codebase lint on merge catches issues a diff-only PR lint might miss.
  • Named reviewers means accountability is unambiguous.

Cons

  • Skipping tests post-merge assumes merge is always fast-forward — a rebase-with-conflicts merge could theoretically ship untested code.
  • No automated check that the merged branch is still green against latest develop before publish.

Use cases this fits

  • Teams merging frequently in small increments, where fast-forward merges are the norm.
Stage 05 · Continuous Deployment

The Freestyle CD job, step by step

Triggered only after the upstream CI job reports stable. Workspace is wiped, a fresh Python venv is built, and a deploy script drives kubectl against the resolved namespace/context.

05 / 08
triggerupstream job stable
→
env setupwipe workspace
→
venvpython3 -m venv .venv
→
deploydeploy_to_k8s.py
→
notifyHTML email
Build Steps — Freestyle shell block
set -e  # fail fast

echo "=== Creating virtualenv ==="
python3 -m venv .venv

echo "=== Activating virtualenv ==="
. .venv/bin/activate

echo "=== Installing dependencies from requirements.txt ==="
pip install --upgrade pip
pip install -r requirements.txt

echo "=== Running deploy script for app ${app_name} on env ${target_build_env} ==="
python -m scripts.deploy_to_k8s \
  --app "$app_name" \
  --env "$target_build_env" \
  --namespace "$kubectl_namespace" \
  --context "$kubectl_context"

echo "=== Deactivating virtualenv ==="
deactivate
echo "=== Build step finished OK ==="

Pros

  • set -e plus a fresh venv/workspace every run means no silent partial-failure or stale-dependency deploys.
  • "Trigger only if build is stable" is exactly the right gate — a red CI build can never cascade into a bad deploy.
  • HTML email notification on every deploy gives on-call visibility without opening Jenkins.

Cons

  • "Success" means the script exited 0 — nothing here confirms the pod actually reached Ready post-deploy.
  • Freestyle job logic isn't code-reviewed the way a Jenkinsfile change would be; it lives only in Jenkins' own config history.
  • Credential handling for the Artifactory/kubectl secrets isn't visible in the build step — worth an explicit audit (see Security).

Use cases this fits

  • Small number of services where a Python deploy script wrapping kubectl is simpler than adopting a full GitOps tool.
  • Environments where "who deployed what, when" via email is sufficient audit trail.
Worth adding

A post-deploy health check — kubectl rollout status deployment/$APP_NAME -n $NAMESPACE --timeout=120s — as the last build step, so "green" in Jenkins actually means "serving traffic," not just "script didn't crash."

Stage 06 · Security & governance

What's enforced, and what to verify

Branch protection and artifact immutability are structurally enforced. Credential handling is the one area the docs describe well but the running config doesn't visibly confirm.

06 / 08

Enforced today

  • Mandatory PR review on protected branches — no direct commits.
  • Artifact immutability via Artifactory once pushed.
  • Manual approval gate before any production deployment.

Unverified from the config

  • Whether the Artifactory/kubectl credentials are bound via Jenkins Credentials Manager (withCredentials) or sit as a plaintext var.
  • Whether build logs ever echo a secret value — easy to introduce accidentally with set -x or a stray echo $PASSWORD.

Quick audit checklist

  • Grep Freestyle job configs for any credential referenced by literal value, not by ID.
  • Confirm Jenkins "Mask Passwords" plugin (or equivalent) is active on CD jobs.
  • Rotate the Artifactory service account password once the audit is done, regardless of outcome.
Risk pattern to check for
kubectl create secret docker-registry svc-secret \
  --docker-password=MyActualPassword123 \  # ← literal value in job config
  --namespace=it-ex-automation-dev
What it should look like
withCredentials([string(
  credentialsId: 'artifactory-svc-password',
  variable: 'ARTIFACTORY_PASSWORD')]) {
  sh 'kubectl create secret docker-registry svc-secret \
       --docker-password=$ARTIFACTORY_PASSWORD ...'
}
Stage 07 · Naming the shape

Patterns underneath the pipeline

None of this is unusual — but naming it precisely makes it easier to reason about and to onboard someone new onto.

07 / 08
PatternWhere it shows up
Trunk-adjacent GitFlowfeature → develop → release, fully PR-gated, no direct commits.
Branch-conditional pipelineOne Jenkinsfile validates on PR branches, publishes on develop — same file, different behavior.
Job-chaining triggerCD Freestyle job fires via "build after other projects," decoupling CI and CD into two Jenkins job types.
Immutable artifact promotionOne Docker image built once; environment behavior injected at deploy time, not baked in per-env rebuilds.
Environment-as-job-duplicationEach environment is its own Freestyle job with hardcoded literals, instead of one parameterized job.
Ephemeral build environmentFresh venv and wiped workspace every CD run — no shared mutable state between builds.
Push-based notificationEmail fires on every deploy rather than requiring someone to poll Jenkins for status.
Stage 08 · What to change, in order

Improvement roadmap

Ordered by impact-to-effort, not by how the sections above happened to appear.

08 / 08
01

Parameterize the CD job, config lives in the repo high impact

Replace one hardcoded Freestyle job per environment with a single job that reads namespace/context from an envs.yaml committed alongside the service. Turns "add an environment" into a one-line, reviewable diff.

02

Move CD to pipeline-as-code medium effort

Bring the CD Freestyle logic into a Jenkinsfile (or a shared Pipeline Library step), so deploy-script changes get the same git history and code review as everything else in this guide's CI section.

03

Audit credential handling do first, low effort

Confirm every secret in a CD job is bound via Jenkins Credentials Manager, not a literal value in the job config or a var that could get echoed into build logs.

04

Add a post-deploy health gate medium effort

End the CD build step with kubectl rollout status (or an HTTP health check) so a green Jenkins build actually means the workload is serving traffic, not just that the script exited cleanly.

05

Name a backup approver low effort

Add at least one additional reviewer (or a role-based Bitbucket group) so merges never block entirely on two people's availability.

06

Consider a pull-based deploy model longer-term exploratory

Even on-prem, an in-cluster agent watching a manifests repo (Argo CD-style) reduces the blast radius of Jenkins holding cluster-admin-equivalent credentials for every namespace it deploys to.
